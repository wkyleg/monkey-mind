Got it — since your headband sits at the front of the head, SSVEP (Steady-State Visual Evoked Potentials) is a really practical way to do simple “left/right” game control.
How SSVEP works
When someone looks at a flickering stimulus (like an LED or a square on the screen) that blinks at a fixed frequency (e.g. 10 Hz or 12 Hz), the visual cortex naturally produces an oscillation at the same frequency.
Even though your electrodes are at the front, EEG can still pick up some of this rhythmic activity volume-conducted from the occipital cortex. In practice, front-headband SSVEP works best if the person is actually looking at the flickering stimuli on the screen, so you don’t need back-of-head electrodes.
You can assign one flicker frequency to “left” and another to “right.” The player looks at the target they want, and the system detects which frequency dominates in the EEG.
Coding pipeline (simplest working loop)
Stimulus presentation
On your game screen, place two visual targets: left and right.
Make the left box flicker at 10 Hz (change brightness every 50 ms), right box flicker at 12 Hz (every ~41.7 ms).
EEG acquisition
Stream raw EEG data from your ADS1299 → Raspberry Pi setup (as in your Elata EEG pipeline
Elata Engineering White Paper-2
).
Pick 2–4 frontal channels for simplicity.
Preprocessing
Bandpass filter around the target frequencies (e.g. 9–11 Hz and 11–13 Hz).
Optionally apply notch filter to remove mains interference (50/60 Hz).
Frequency detection
Simplest method: compute FFT (Fast Fourier Transform) or Welch periodogram on a short window (e.g. 1–2 seconds).
Compare the power at 10 Hz vs. 12 Hz.
Whichever band has higher power → interpret as left or right intent.
More robust: use Canonical Correlation Analysis (CCA), which compares EEG with reference sinusoids at 10 Hz and 12 Hz to see which matches better.
Game control
Every 1–2 seconds, output a command (“left” or “right”) to your Pong paddle or whichever game engine you’re using.
Keep smoothing decisions (e.g. majority vote across last 3 windows) to reduce jitter.
Why this is the “simplest thing that works”
Only needs two frequencies = binary control.
FFT or CCA are trivial to implement in Python or Rust with existing libraries (NumPy, SciPy, or Rust crates).
Front headband is good enough because the user is looking at the flickering box → signals propagate forward.
No training data is required (unlike motor imagery methods), so you can demo it fast.

5) Input System (BCI-Ready)
5.1 Canonical Input Model
Gameplay logic consumes intent, not device events.
export type PlayerIntent = {
  moveAxis: number;        // -1..1 (left/right)
  confirm: boolean;        // e.g., start, accept
  cancel: boolean;         // pause/back
  calm: number | null;     // 0..1, optional now
  arousal: number | null;  // 0..1, optional now
};
Keyboard maps to moveAxis via left/right (or A/D); confirm via space/enter; cancel via esc.
Touch maps via swipe left/right; tap for confirm; two-finger tap for pause.
Gamepad maps via stick/dpad.
5.2 Input Providers
InputManager aggregates providers:
interface InputProvider {
  update(dt: number): void;
  getIntent(): Partial<PlayerIntent>;
}
Providers:
KeyboardProvider
TouchProvider
GamepadProvider
DebugSignalProvider (sliders for calm/arousal; dev only)
Later: BCIProvider (SDK adapter)
Conflict resolution:
Prefer strongest non-zero moveAxis from any provider.
confirm/cancel are OR’d.
calm/arousal choose highest-confidence non-null signal (BCI later); debug if present.
